# Este archivo es CRÍTICO en Python DevOps. Define todas las DEPENDENCIAS EXTERNAS
# que el proyecto necesita para ejecutarse.
#
# CONCEPTO: "Dependency Management" = Control riguroso de qué versiones de librerías
# usa el proyecto. Si cada developer usa versión diferente, el código "funciona aquí
# pero no allá" (nightmare de debugging).
#
# ¿QUÉ ES UNA DEPENDENCIA?
# Son librerías externas que escribió alguien más, que nuestro código usa.
# Nosotros no escribimos pytest, flake8, etc. Las descargamos e instalamos.
#
# ¿POR QUÉ NO INCLUIRLAS EN EL CÓDIGO?
# Porque ocuparían MEGAS de espacio, y habría duplicados si 10 librerías usan la
# misma dependencia. Python instaló un sistema: Package Index (PyPI), donde miles
# de developers suben librerías. `pip install` las descarga automáticamente.
#
# ==================================================================================

# ==========================
# TESTING - Marco de Pruebas
# ==========================
# pytest==7.4.3
#
# PYTEST es el estándar para testing en Python (industria recomendado).
# ¿QUÉ HACE?
# - Busca archivos con nombre test_*.py o *_test.py
# - Busca funciones con nombre test_* dentro
# - Ejecuta cada una y reporta resultados (PASS/FAIL)
# - Permite assertions simples: assert x == y (vs assert x.equals(y) en otros)
#
# ¿POR QUÉ EXACTAMENTE ESTA VERSIÓN (7.4.3)?
# Diferentes versiones = diferentes comportamientos, diferentes APIs, diferentes bugs.
# Si usamos 7.4.3 hoy, en 6 meses cuando alguien nuevo clona el repo, obtiene 7.4.3
# (no 8.0.0 con cambios incompatibles).
#
# ¿QUÉ SIGNIFICA "=="?
# Version pinning = EXACTAMENTE esta versión, no otra.
# - pytest==7.4.3 → Solo 7.4.3 (estricto)
# - pytest>=7.4.3 → 7.4.3 o mayor (flexible, riesgoso)
# - pytest~=7.4 → 7.4.x pero no 8.0.0 (medio flexible, seguro)
#
# MEJOR PRÁCTICA: Producción usa == (exacto). Desarrollo usa >= (flexible).
pytest==7.4.3

# ===================================
# COVERAGE - Medición de Cobertura
# ===================================
# pytest-cov==4.1.0
#
# Es un PLUGIN para pytest que mide cobertura de código.
# ¿QUÉ ES COBERTURA?
# Métrica: "¿Qué % de mi código es ejecutado por tests?"
# Ejemplo:
#   Código total = 1000 líneas
#   Código testeado = 850 líneas
#   Cobertura = 85%
#
# ¿POR QUÉ IMPORTA?
# - 0% cobertura = ningún test (código completamente untested, muy riesgoso)
# - 50% cobertura = mitad untested (problemas escondidos en la otra mitad)
# - 80%+ cobertura = confianza razonable (bugs improbables)
# - 100% cobertura = teorico (algunos paths imposibles, no vale la pena)
#
# OBJETIVO REALISTA: 70-80% cobertura
# (20-30% es code de error handling raro, no vale gastar tiempo testeándolo)
#
# ¿QUÉ GENERA?
# - Reportes XML (para herramientas como Codecov)
# - Reportes HTML (para revisión visual, color rojo = no testeado)
# - Terminal output (resumen rápido)
pytest-cov==4.1.0

# ================================
# LINTING - Análisis de Código
# ================================
# flake8==6.1.0
#
# FLAKE8 analiza código SIN EJECUTARLO (análisis estático).
# ¿QUÉ DETECTA?
# - Errores de sintaxis (import que no existe, variable no definida)
# - Código muerto (variable definida pero nunca usada)
# - Mal estilo (inconsistencias con PEP 8 = estándar Python)
# - Complejidad alta (función con 50 if/else anidados)
#
# EJEMPLO DE PROBLEMAS DETECTADOS:
# ✗ x = 5; print(z)  → 'z' no definida (flake8 lo ve)
# ✗ def func(a, b, c, d, e...): → Demasiados parámetros (complejidad)
# ✗ x=5; y = 6  → Inconsistencia de espacios alrededor de =
# ✗ import unused_module  → Import nunca usado
#
# ¿POR QUÉ IMPORTA?
# - Previene bugs tontos (typos en nombres)
# - Asegura consistencia (todo el equipo code igual)
# - Prepara para refactoring (código limpio es más fácil de cambiar)
# - Educativo (novatos aprenden convenciones Python)
flake8==6.1.0

# ==================================================================================
# ¿CÓMO SE INSTALA ESTO?
# ==================================================================================
# Comando: pip install -r requirements.txt
#
# Esto hace:
# 1. Lee requirements.txt
# 2. Para cada línea no comentada (pytest==7.4.3, etc):
#    a. Descarga esa versión exacta desde PyPI
#    b. Instala las dependencias de esa dependencia (recursivo)
#    c. Verifica que no hay conflictos entre versiones
# 3. Todo está listo para usar
#
# ¿CUÁNDO ACTUALIZAMOS?
# Nunca, a menos que:
# - Un bug crítico de seguridad sale (actualizar a patch, ej 7.4.3 → 7.4.4)
# - Feature nueva es crítica (actualizar a minor, ej 7.4 → 7.5)
# - Hay incompatibilidad con Python nueva (rara, pero puede pasar)
#
# PROCESO SEGURO DE ACTUALIZACIÓN:
# 1. En rama feature: pip install --upgrade pytest==8.0.0
# 2. Correr TODO los tests con versión nueva
# 3. Si TODO PASA: actualizar requirements.txt (commit)
# 4. Si FALLA: revertir, esperar a versión más estable
#
# ==================================================================================
# ¿CUÁL ES LA DIFERENCIA CON setup.py / pyproject.toml?
# ==================================================================================
# requirements.txt = DEPENDENCIAS PARA EJECUTAR EL PROYECTO
# setup.py / pyproject.toml = DEPENDENCIAS PARA DESARROLLAR + METADATOS DEL PROYECTO
#
# Ejemplo:
# requirements.txt:
#   pytest  ← solo testing
#   flake8  ← solo linting
#   numpy   ← la app realmente usa numpy
#
# pyproject.toml:
#   dependencies: [numpy]  ← la app real necesita
#   dev-dependencies: [pytest, flake8]  ← developers necesitan
#   name: axanet-cliente-manager
#   version: 1.0.0
#   author: DevOps Team
#
# En este proyecto solo tenemos requirements.txt (suficiente para educación).
# En producción real, una app usa setup.py y requirements-dev.txt (testing separado).
#
